Project Description

Overview

Topic

Our project is an interactive program that informs and then tests the users on basic cyber security practices. We plan to build a prototype of this program that users will be able to download onto a computer and then use to increase their knowledge of security. We have chosen to build this as a program instead of a webpage to allow it to be more modular. We will build this program using Python, and Pyqt5, a module that streamlines User Interface (UI) design and creation. The focus of the UI will be that it is simple and intuitive to ensure that inexperienced users will be able to successfully navigate the program without much trouble.

The lessons in the program will be designed using information that is drawn from credible sources, such as peer-reviewed journals, to ensure that the lessons are both accurate and helpful. Furthermore, the program will test users based on what lessons they have completed, with this feature assisting in finding the gaps in their knowledge, and areas where they might choose to repeat a lesson. To increase user retention, and achievement feature is used to ‘gamify’ the program and motivates users to come back to try and achieve better scores on the tests.

Motivation

This project is useful because it will be aimed at teaching cyber security practices in an easy-to-understand manner to allow users of almost any level of computer knowledge to be able to use and understand the information contained in the program. “Attacks on the Internet of Things devices will increase rapidly due to hypergrowth in the number of connected objects, poor security hygiene, and the high value of data on IoT (Internet of Things) devices” (Sciencedirect.com, 2015, page 26 para. 2). This rise in attacks signifies a need to educate people who lack the necessary knowledge about tech and inform them of how they can protect themselves. This I especially imperative currently as a society become more entwined with technology, and inexperienced users are expected to interact with it daily. Working on this project would show future employers that the team member has a baseline understanding of python programming and user-centred design, can work in team environments and knows cyber security practices.

Landscape

With the entwinement of individuals' financial and personal information into the internet, many organizations have resources for the public to help educate them on cyber security. One such resource is “cyber.gov.au”, a federal government website run by the Australian Cyber Security Centre (ACSC). It provides information about cyber security to both personal users and commercial groups and has a quiz for users to see what level of knowledge the user has. This holds some similarity to our project, however, it also differs greatly, as it is a webpage as opposed to a program, and the testing component is more so about establishing the first-time user’s baseline knowledge as opposed to reinforcing lessons. Other than this, the market is remarkably bare, with most resources focused on people hoping to have a career in cyber security

Aims

This project aims to help enlighten people on the threats on the internet by teaching people all skills with technology. This will be done through a learning system with a quiz to consolidate the user's knowledge. The learning system will consist of both imagery and notes to read that are at the level that anybody can read the information no matter what level they are at with the imagery, this would also have to be readable so that it is not too complex for the average user but not so dumbed down that people get bored of the information that they are reading. The aim is for people to learn about the topics of online safety, privacy, social media/communication safety, device safety, and password safety so that they can make sure that they are taking the correct steps online to keep themselves safe from any dangers that these poses. The goal is not to protect but rather for people to understand how to be safer on the internet and have a good understanding of what they are doing and what they should be avoiding.

A goal for achieving this is to make a basic prototype that shows what the goal and potential of the application are. Showing the potential of the application would then create interest in the application, especially for those who are looking out for new and upcoming applications or something that could be of use to them. This would help with people being able to give some pre-testing input so that the application can be more directed to them.

A testing phase would help with this goal as it would allow for relevant changes to be pointed out and fixed for the future. This also would be able to bring in the target audience for the first time to get their true opinions on how they feel about it. This would push for the application to allow for more inclusivity as it should be for all types of people and the more people that test would be able to show anything that is missing. It would also help with knowing whether the quizzes help with people's understanding of the topics or if the degree of difficulty needs to be changed on them.

Plans and Progress

Our project is fundamentally designed around helping people keep their devices secure and informing them on how to identify and circumvent scams and other types of online misconduct. Additionally, we aim to provide detailed information for those who wish to refresh their knowledge on these security practices as well as people who have less than adequate general computer knowledge. Due to the large variety of users, we want to be able to have an effective experience with our project, all information within the project must be communicated in an easy-to-understand manner. We plan to implement this through simplified explanations on each topic and a focus on the user experience, eg. how the project looks and functions.

Our project will achieve its purpose through the use of a program. The program is written in python and uses the modules PyQt5 and XML.etree. We chose to use Python for our program because it was the language our team had the most experience with, making it more time-efficient to learn the specifics that we would need for this project. We chose to use PyQt5 as it’s a highly customizable and robust UI module. It also includes a designer that can be used to visualize the entire program’s UI as we make it, resulting in a significantly shorter development period. We used XML.etree due to it being included in every Python install and thus, well documented as an efficient XML parsing module. \XML files were chosen to store basic data for each page and section such as the section name, the body (main text) of each page, each question relating to that section, and whether the section has been completed by the user. Storing this data outside of the main program, allowed us to implement some modularity into it. So instead of hardcoding each module, section and page into the program, we can cut down the amount of redundant coding significantly by making a shorter piece of code parse through the XML file for this information and using whichever pre-made layout, images or text that the XML file specifies.

Our program is split into three distinct areas. Firstly, we have the teaching area. This area of the program will include categorized modules, within which there will be separate sections and pages. Each module will cover a general security topic including online safety, privacy, social media/communications, device safety, and password safety. The teaching page communicates its information through short paragraphs of text with images displayed sporadically. Once an entire section is read, it will be marked as complete in the XML file.

The second area of the program is the testing area. This part of the program gives the user several multiple-choice questions based on which sections have been marked as complete in the XML file. Once all the questions have been completed, the user is given their score and sent back to the main screen.

The final area of the program is the achievement area. This area will add a sort of gamification to the program to keep the user’s attention and give them some goals to strive for. This area will include various achievements that the user can earn in other areas of the app (eg. completing a certain amount of questions without getting one wrong). Whenever a user earns an achievement, they will be notified through a pop-up, regardless of where they are in the program and if they go to the achievement area, they will be able to identify which achievements they have and have not completed.

We began this project by splitting it into two groups. The first group were given the role of a programmer while the second was given the role of UI designers. The programmers began their work by researching PyQt5 and learning that python module through YouTube videos, the Qt forums, StackOverflow and lots of trial and error within testing documents. The UI design team began by working on some basics for the wireframe and then presenting it during a group meeting. After this, they made changes where necessary and continued working.

As the programming group progressed with the prototype of the program, we quickly noticed that due to time constraints and a lack of ability, we weren’t going to complete an entire working prototype. Due to this, we focused the development on the major areas of the program and split it all up into a few separate files for everyone to easily work on. The prototype file is the achievement area of the program. Here we used PyQt5’s Qframes as a visual identifier for which achievements have been obtained. The Qlabels and QPushButtons on the program have stylesheets applied to make them resemble the UI created by the UI design team. To simulate how the achievements would work, we placed some example buttons in the bottom-left of the UI, these buttons are purely for testing purposes and won’t be included in the final program. Once the program has noticed an achievement has been obtained, it changes the colour of the Qframe to green, the border will be removed through stylesheets, and a Qmessagebox is displayed that mentions the achievement to the user.



For the teaching part of the program, we recreated the UI that the UI design team made in their wireframe once again through stylesheets. We placed a Qlabel at the top left of the program to indicate when an input is received from any section of the UI. The program uses a Qscrollarea to allow for scrolling within the program. In contrast to the other prototypes, this one uses Qlayouts to make each UI element relative to each other (similar to how you would use divs and floats in a website) This was required to get the Qscrollarea to work correctly. Upon scrolling down there are two buttons displayed that update the output Qlabel.



For the testing section of the program, we mainly focused on functionality instead of stylesheets for appearance. A question is displayed as a Qlabel in the top area while each of the Qpushbuttons changes the output text Qlabel in the middle.



For the XML prototype of the program, we created an example XML document that would represent what we would use for the actual program



We then wrote a script to parse all the data we require for the functioning of the program. This function used the XML.etree.elementtree module to parse the data.



The script outputs all of the data by printing it into the console. In the future development of our program, this data will be shown through Qlabels, QlistWidgets other UI elements. In our prototypes that focused on PyQt5, we made a base to build each prototype. This is shown below.



We start by importing all the modules that will be used. Next, we create a python class as the Qmainwindow for the UI. The next piece of code changes the DPI scaling so it displays correctly on most devices (since some laptops have unusual DPI scaling, it can disrupt how the UI is displayed). After this, we announce a function for initializing the UI. Inside this function, a UI file created in QtDesigner is loaded and some short lines of code specify a function that connects to an element of the UI. These functions are defined below, and in this case, the Qlabel named lblOutput has its text changed. Finally, an instance of the program is created as a Qapplication and the app is executed.

While the programmers partook in developing the backend of the program, the UI design team took up the task of working on a minimalistic front end. The decision to keep the program minimal was already established before the start of the assignment due to the concern that the program may come across as overwhelming for its target audience. This makes it easier to navigate and adds benefit to the programming team by making it easier to build. The design is made through Figma, a design tool, using standard laptops. Their team of two hosted one on one meetings held through Discord to make communication easier. As part of the brainstorming process, many ideas were brought up and then discarded when creating the basic mockup for the user interface. Plans for the mockup of the program were made before the actual process through low fidelity sketches with Adobe Photoshop to reduce the amount of time needed to correct mistakes. In Photoshop – which was streamed by a group member - the design team went through all the necessary design elements, including but not limited to the basic layout for the main pages and sections, where the buttons should go, the color scheme, and even the spacing of the features as well as accommodating for those with disabilities by incorporating accessibility options into the menu.

During the actual process of developing the mockup, some concerns were raised which were missed through the planning process; the design team strayed from some parts of the plan to make necessary adjustments. They felt that the blue background of the program did not provide enough contrast against other elements since they are trying to accommodate those with visual impairments while keeping the program as minimalist as possible. The blue was then adjusted to be darker and richer, which contrasted well with the chosen shade of yellow that was used for the buttons, as well as the white for other elements such as the icons, menu, and text.

The basic mockup was finished on Monday and was ready to be presented to the rest of the team. After a brief explanation of what the basic mockup looks like, a member of the programming team provided constructive criticism to improve the interface by suggesting navigation buttons to give the users the freedom to move around, as well as a ‘Scroll Down’ feature with a downwards arrow to show the users that there is additional information stored below the page they are currently seeing. The feedback was then considered, and the design team promptly added them. Only then did the design team realize that the sizes of the buttons were too large and took up the whole page which was then shrunk. They refrained from adding a login function after a quick read of the project plans; users need to store their progress locally through XML files, and if a login feature were added, the group would have to host servers which goes against what was established in the plan. Some problems required last-minute adjustments, with team members remembering to add necessary features, like the option to quit the program. Furthermore, the Test Your Knowledge section slightly differed from the testing section, as it was marked with a flag to see how much the user has learned by pulling random questions from random sections, they have gone through compared to the testing section where they’re asked about the topic they are in. After that, the design team kept working until the interface was polished and refined.

The ‘shut down’ button is included due to the application of recognition over recall, an established usability heuristic so that users will recognize that it’s the button for when they want to quit the program.

Once they are done reading through the learning section, they are then taken to the testing section where they are then assessed. Risks

Our project idea concerns the arising problems of cyber threats that crawl their way into the digital world. We aim to educate those who are unaware of basic cybersecurity practices and fall victim to malicious software. However, several risks come with every project that is being built into fruition.

Our main priorities for the project involve a minimalistic, easy to comprehend user interface. The target audience is those who are not as well versed with technology and need to learn how to navigate through cyberspaces safely. A problem with the front end is that some of the features may come across as niche or too advanced since it is difficult to put ourselves in our target user’s shoes, we’re entirely unsure how they would receive the program and its aesthetics, which might lead to our users disengaging with our program.

Another problem that arises is the struggle of not being proficient in the tools needed to build the program. Our group members went through crash courses to learn PQt5 for the backend. With the lack of proficiency in the tools, some areas of the program will be relatively more difficult to explore and bring from the ground up.

Bugs may also crawl through the program despite testing. While the backend may not have problems initially, some user interactions may cause errors in ways we have not anticipated. This can only be resolved through reports and updates in trying to find solutions to the problem.

Group Processes and Communications

Our Group is expected to do a hybrid communication both in-person and online. This can take place at a venue that we have agreed on such as a library, cafe, or a private lecture room on campus. It is also expected that our online communication will always take place via MS teams, usually every Monday at 5 pm or whenever our group members are available and whenever they cannot attend in person within our time frame accordingly, though some group members especially the UI design team communicate through external applications such as Discord. When circumstances arise such as one of our group members not responding to communications, we should contact them via other university-related avenues such as their student email and ask them how they could continue to contribute to our team, but if this does not work and a week passes with absolutely no communication, we would let our coordinators know and see what we can do about that team member.

Skills and Jobs

Cyber Security Expert / Researcher


A Cyber security expert would be needed to further develop this project in the long term. This role would largely be focused on collating information about cyber security practices and creating lessons out of that information that would be displayed to users. This team member would have to be well versed enough in the topic to be able to simplify and explain cyber security concepts to a user who has little prior knowledge on the subject. Ideally, this team member should also have a degree of competency in visual design, as they would be tasked with creating visual aids, such as graphs, to assist in the lessons. Furthermore, they must have some experience in programming as they would be tasked with inputting the lessons into XML files, as well as editing those lessons that are already in that format, should the need arise.

User Interface Designer


To finish designing the program, a user-centred designer would be needed. This role would primarily be involved with the creation of designs for further features and pages that may be added in the program’s future. To achieve this, they must be fluent in technical design skills, and have experience with designing mockups and prototypes of programs, using tools such as Figma. Furthermore, they would require a strong understanding of heuristic principles and common design patterns that ensure the creation of an intuitive user interface. Through working in collaboration with the user tester, they must beable to further improve the designs using the feedback given.

Programmer


With functionality being such a need, a programmer would be required to complete this program. The prototype in its current form is for demonstration purposes only, and as such, it would be the programmer’s role to ensure that the program becomes functional for users. To achieve this goal, they must be well versed in Python, as well as the PyQt5 and XML.etree modules. In addition to this, the programmer would need to help innovate new features for the program and be able to fix bugs and errors should they arise. For this, they would need the capacity for innovative thinking and problem-solving. They should also be experienced in working within a collaborative work environment.

User Tester


Due to the program’s target audience being those who lack experience with online technologies, it is imperative to gather user feedback to make improvements and add features to improve the experience. This would be achieved by a user tester. This role would include user testing of the program, through various means, such as lab tests and experience sampling, to gather user feedback. They must have knowledge of how to gather relevant user groups for testing. They would also be required to not only to have the technical skills to set up lab tests and create experience sampling questionnaires, but also to have a firm grasp of the ethics of user testing, to ensure that all tests are legal and safe for the subjects. After gathering the information from the observations and tests, the user tester must be able analyse and present the findings to the rest of the team, in order to facilitate a data driven approach to the improvement of the program.
